# Battery Workforce Research Repository

## Project Overview

This repository supports the Volta Foundation's "2026 State of Industry: How Many Workers Are Needed for Battery Manufacturing?" research initiative. The goal is to quantify total battery manufacturing workforce demand per GWh and project global workforce needs through 2050.

## Repository Structure

```
batt-workforce/
  Plan.md                       # Living project roadmap and task checklist
  data/
    raw/                        # Primary research files (PDFs, spreadsheets) — human-provided
    processed/                  # Agent-extracted structured data
      sources.yaml              # Registry of all primary sources + processing status
      sensitivities/            # One subfolder per identified sensitivity
        automation/
        energy_density/
        market_share/
        manufacturing_productivity/
        policy_scenarios/
  site/                         # Combined report + dashboard (deployed via GitHub Pages)
    index.html                  # Landing page with navigation
    css/styles.css              # Shared styles
    js/dashboard.js             # D3.js interactive dashboard logic
    data/                       # Generated JSON consumed by the dashboard
      labor_intensity.json
      global_jobs.json
  scripts/
    generate_dashboard_data.py  # Reads processed/ CSVs, outputs site/data/ JSON
    process_source.py           # Template for processing a new raw source
    requirements.txt
  .github/workflows/deploy.yml  # GitHub Pages deployment
```

## Key Concepts

- **Labor Intensity (View A)**: Workers per GWh vs. time from factory start. Shows how workforce needs decline as factories mature. Data aggregated from multiple primary sources; curve-fitted to derive a master scaling equation.
- **Cumulative Global Jobs (View B)**: Projected battery workforce through 2050, parameterized by regional market share, growth rate, and policy scenarios.
- **Sensitivities**: Variables that significantly impact total workforce numbers. Each sensitivity lives in `data/processed/sensitivities/<name>/` with CSV data and metadata YAML.

## Working with This Repo

### Adding a new primary source
1. Place the raw file in `data/raw/`
2. Update `data/processed/sources.yaml` — mark the source as `provided: true`
3. Process the source: extract relevant quantitative data into CSVs under the appropriate `sensitivities/` subfolder
4. Run `python scripts/generate_dashboard_data.py` to regenerate dashboard data
5. Commit and push — GitHub Pages auto-deploys

### Updating the dashboard
- Dashboard source lives in `site/` — HTML, CSS, JS
- Data is in `site/data/*.json` — generated by Python scripts from processed CSVs
- D3.js is used for all interactive charts
- After changing data or scripts, run `python scripts/generate_dashboard_data.py`

### Processed data format
- **CSV files**: machine-readable tabular data (one per dataset)
- **metadata.yaml**: accompanies each CSV with provenance, assumptions, units, and sensitivity category
- **Mathematical assumptions**: documented in metadata YAML under `assumptions:` key — includes curve fit parameters, extrapolation methods, confidence intervals

## Development Commands

```bash
# Regenerate dashboard data from processed sources
python scripts/generate_dashboard_data.py

# Process a new raw source (template)
python scripts/process_source.py --source <source_name> --input data/raw/<file>

# Local preview (any static server works)
python -m http.server 8000 --directory site/
```

## Conventions

- Commit messages should be descriptive and reference the task being accomplished
- All quantitative claims in the report must trace back to a source in `sources.yaml`
- Sensitivity parameters should be documented with units, ranges, and source references
- When processing sources, prefer extracting too much data over too little — we can filter later
